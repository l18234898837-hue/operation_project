# 光伏智能问答系统二人模块化协作执行规范

> 版本：v1.0  
> 适用阶段：第一阶段（文档知识库 RAG Web MVP）  
> 适用对象：两名开发者、项目协作人员、使用 Codex 辅助开发的成员  
> 核心原则：按模块顺序推进，两人共同完成同一个模块；模块内部每日实时确认文件边界和任务边界。

---

## 1. 文档目的

本文档用于将第一阶段开发过程拆成可执行的模块流程，并规定两名开发者如何在同一个模块内并行协作、如何使用 GitHub 管理代码、如何避免文件冲突、如何判断一个模块已经完成。

第一阶段目标不是一次性做完整 AI 助手，而是优先跑通一个稳定、可信、可追溯的文档问答闭环：

- 管理员可以上传文档
- 文档可以解析、清洗、切块、向量化、入库
- 用户可以在 Web 页面提问
- 系统可以基于文档证据回答
- 回答必须带来源引用
- 信息不足时可以追问
- 无依据或超范围时可以拒答
- 日志、未命中问题、trace_id 可以追踪

---

## 2. 协作模式总原则

本项目采用“模块化共同推进”模式，而不是一开始就完全拆成前端和后端两条独立路线。

推荐协作模式：

```text
一个模块一个模块推进。
每个模块两个人共同负责最终结果。
模块内部仍然划分当天的文件边界和实现边界。
共享文件指定一个主改人，另一个人 Review。
模块完成后通过一个模块 PR 合并到 main。
```

这个模式适合当前阶段，原因是：

- 两个人可以保持对整体系统的共同理解
- 每个模块都能形成一个可验证结果
- 接口、数据结构、页面行为可以在模块内及时对齐
- 不容易出现前后端各写各的、最后才发现无法联调的问题
- 项目早期可以快速发现架构、接口和范围问题

## 3. GitHub 分支模型

## 3.1 主分支

### `main`

`main` 是稳定分支，只保存已经完成并通过基本验证的模块结果。

规则：

- 不直接在 `main` 上开发
- 不直接 push 到 `main`
- 所有模块通过 Pull Request 合并
- `main` 上的代码应该随时可以启动或至少保持结构完整

## 3.2 模块分支

每个模块建立一个模块分支，两个人都在该模块分支上协作。

命名格式：

```text
feature/module-编号-模块英文名
```

推荐分支：

```text
feature/module-01-bootstrap
feature/module-02-frontend-pages
feature/module-03-api-schema
feature/module-04-document-manage
feature/module-05-document-ingest
feature/module-06-retrieval-qa
feature/module-07-history-logs
feature/module-08-evaluation-acceptance
```

## 3.3 临时个人分支

如果某个模块内改动较大，或者两个人会频繁改到相邻文件，可以在模块分支下再开个人临时分支。

命名格式：

```text
work/姓名或昵称/模块编号-任务名
```

示例：

```text
work/a/module-03-upload-api
work/b/module-03-upload-page
```

临时分支使用规则：

- 临时分支只合并回当前模块分支
- 不直接合并到 `main`
- 当天或两天内尽量合并，避免分支长期漂移
- 合并前先从模块分支同步最新代码

如果项目初期改动不复杂，可以先不使用个人临时分支，直接两个人共同提交到模块分支。

## 4. GitHub Issue 管理规则

## 4.1 一个模块对应一个主 Issue

每个模块必须建立一个主 Issue，用于描述该模块的目标、范围、任务列表、完成标准和风险。

Issue 标题格式：

```text
[module-编号] 模块名称
```

示例：

```text
[module-03] 文档上传与管理闭环
```

## 4.2 模块 Issue 内容模板

```markdown
## 模块目标
本模块要跑通什么能力？

## 输入条件
- 依赖哪些前置模块？
- 需要哪些样本文档？
- 是否依赖外部模型或数据库？

## 本模块包含
- 功能点 1
- 功能点 2
- 功能点 3

## 本模块不包含
- 暂不做的内容 1
- 暂不做的内容 2

## 今日边界记录
### YYYY-MM-DD
- A 负责：
- B 负责：
- 共享文件主改人：
- 今日最小目标：

## 接口或数据约定
- 接口：
- 请求：
- 响应：
- 状态枚举：

## 完成标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 验证方式
- 如何启动？
- 如何测试？
- 用哪个样本文档或问题验证？

## 风险与阻塞
- 风险 1
- 风险 2
```

## 4.3 子任务 Issue

如果模块较大，可以创建子任务 Issue。

示例：

```text
[module-03][backend] 实现文档上传接口
[module-03][frontend] 实现文档上传页面
[module-03][docs] 补充文档上传接口契约
```

子任务 Issue 完成后关联到模块主 Issue。

## 5. 每日边界确认机制

为了避免同一个模块同一个分支下的冲突，两个人每天开始开发前必须先确认当天边界。

## 5.1 每日开工前 10 分钟确认

每天开始前固定回答 6 个问题：

```text
1. 今天处于哪个模块？
2. 今天这个模块要跑通的最小结果是什么？
3. A 今天主要改哪些目录或文件？
4. B 今天主要改哪些目录或文件？
5. 哪些共享文件今天可能会被改？谁是主改人？
6. 今天结束前要如何验证？
```

## 5.2 边界记录位置

每日边界必须记录在当前模块 Issue 中。

示例：

```markdown
### 2026-06-15
- 今日模块：module-03 文档上传与管理
- 今日最小目标：前端可以选择文件并调用后端上传接口，后端可以保存文件元数据并返回 document_id。
- A 负责：backend/app/api/documents.py、backend/app/services/document_service.py
- B 负责：frontend/src/pages/DocumentManage.vue、frontend/src/api/documents.ts
- 共享文件主改人：A 修改 docs/api-contract.md，B Review
- 今日验证：上传一个 DOCX 文件，页面显示 uploaded 状态
```

## 5.3 共享文件主改人规则

以下文件或目录容易产生冲突，必须指定主改人：

```text
README.md
docker-compose.yml
.env.example
docs/api-contract.md
docs/database-schema.md
backend/alembic/versions/*
backend/app/models/*
backend/app/schemas/*
frontend/src/types/*
frontend/src/api/*
```

主改人规则：

- 同一天同一个共享文件只允许一个人主改
- 另一个人只 Review 或提出修改建议
- 如果必须共同修改，先在 Issue 中拆分具体段落或文件区域
- 共享文件变更必须在 PR 描述里说明

## 6. 提交规范

## 6.1 Commit 信息格式

统一使用：

```text
type(scope): description
```

常用 type：

```text
feat     新功能
fix      修复问题
docs     文档
test     测试
refactor 重构
chore    工程配置或杂项
```

scope 建议使用：

```text
backend
frontend
api
db
docs
qa
ingest
retrieval
logs
```

示例：

```text
feat(backend): add document upload endpoint
feat(frontend): build document upload page
docs(api): add document upload contract
fix(qa): handle empty retrieval result
refactor(db): unify document status enum
```

## 6.2 提交粒度

推荐小步提交：

- 一个提交只表达一个清晰意图
- 不把大量无关改动塞进一个提交
- 能运行或能解释的阶段就提交
- 每天下班前必须 push 当前进展

不推荐：

- 一周只提交一次
- 一个提交同时改后端、前端、数据库、文档且没有说明
- 提交信息只写 `update`、`fix bug`、`1`

## 7. Pull Request 规则

## 7.1 PR 粒度

每个模块完成后发起一个模块 PR。

PR 标题格式：

```text
[module-编号] 模块名称
```

示例：

```text
[module-03] 文档上传与管理闭环
```

## 7.2 PR 描述模板

```markdown
## 模块目标
本 PR 完成了什么模块能力？

## 主要变更
- 后端：
- 前端：
- 数据库：
- 文档：

## 接口变化
- 新增接口：
- 修改接口：
- 删除接口：

## 共享文件变更
- 文件：
- 主改人：
- Review 结果：

## 如何验证
1. 启动方式：
2. 操作步骤：
3. 预期结果：

## 已知问题
- 问题 1
- 问题 2

## 是否影响后续模块
- 是 / 否
- 说明：
```

## 7.3 Review 规则

另一个人必须 Review 后才能合并。

Review 重点：

- 是否符合本模块范围
- 是否改动了未约定的共享文件
- 接口字段是否与文档一致
- 数据库状态枚举是否一致
- 前后端是否可以联调
- 是否有基本错误处理
- 是否留下无法解释的临时代码

## 7.4 合并规则

模块 PR 合并前必须满足：

- 当前模块完成标准全部勾选
- 至少完成一次本地验证
- PR 描述写清楚验证方式
- 共享文件变更已被 Review
- 文档已同步更新

## 8. 冲突预防与处理规则

## 8.1 冲突预防

冲突预防优先于冲突解决。

每日开发前：

```text
git checkout main
git pull
git checkout feature/module-xx-name
git pull
```

开发中：

- 尽量不要长时间不 pull
- 改共享文件前先在群里或当面说一声
- 不同时修改同一个迁移文件
- 不同时修改同一个类型定义文件
- 接口字段变化先改文档，再改代码

## 8.2 发生冲突时的处理顺序

如果出现 Git 冲突：

```text
1. 停止继续写新代码
2. 找到冲突文件
3. 两个人一起确认哪部分保留
4. 解决冲突后立即运行相关模块
5. 单独提交一个 conflict fix commit
6. 在模块 Issue 中记录冲突原因和处理结果
```

冲突修复提交示例：

```text
fix(docs): resolve api contract conflict in module 03
fix(frontend): resolve document api type conflict
```

## 8.3 禁止事项

- 禁止为解决冲突直接覆盖对方文件
- 禁止使用不清楚后果的强制推送
- 禁止在未确认的情况下重写模块分支历史
- 禁止为了省事删除对方刚提交的逻辑

## 9. 模块推进顺序

第一阶段建议按以下模块顺序推进。

## 9.1 Module 01：项目底座

### 目标

建立项目基础结构和本地开发环境，让两名开发者基于同一套目录、同一套依赖、同一套数据库环境继续开发。

本模块不按 A/B 分工推进，而是由两个人共同完成环境初始化。目录结构由项目负责人先创建并推送到代码仓库，同事拉取后按相同步骤配置本地环境。

### 包含内容

- GitHub 仓库初始化
- 项目目录结构初始化
- `backend/` 目录
- `frontend/` 目录
- `docs/` 或正式参考文件目录
- 基础 `README.md`
- `.gitignore`
- `.env.example`
- Python 虚拟环境创建说明
- 后端 Python 依赖文件，例如 `requirements.txt` 或 `pyproject.toml`
- Node.js / 前端依赖安装说明
- 数据库环境说明，例如 PostgreSQL + pgvector
- 数据库连接配置示例
- 基础迁移或建表说明
- 后端 FastAPI 骨架
- 前端 Vue3 + TypeScript + Vite 骨架
- 基础 Docker 或本地启动说明，例如 `docker-compose.yml`

### 不包含内容

- 不做真实 RAG
- 不做完整页面
- 不做复杂权限
- 不做外部 API 接入

### 建议边界

本模块不拆分为“谁负责后端、谁负责前端”，而是先确保两个人都能在本地完成相同环境准备。

推荐执行方式：

```text
1. 项目负责人先创建目录结构和基础配置。
2. 项目负责人推送到 GitHub 仓库。
3. 同事从仓库拉取代码。
4. 两个人分别在自己电脑上创建 Python 虚拟环境。
5. 两个人分别安装后端和前端依赖。
6. 两个人分别启动数据库环境。
7. 两个人分别验证后端、前端、数据库连接是否正常。
8. 两个人共同修正 README 中的环境步骤。
```

共享文件建议由项目负责人主改，同事 Review：

```text
README.md
.env.example
docker-compose.yml
backend/requirements.txt 或 backend/pyproject.toml
frontend/package.json
```

同事只需要拉取仓库并按 README 配置，不需要重新设计目录结构。

### 本地环境要求

两个人都需要完成以下环境准备：

- 安装 Python
- 创建并激活 Python 虚拟环境
- 安装后端依赖
- 安装 Node.js
- 安装前端依赖
- 准备 PostgreSQL 数据库
- 安装或启用 pgvector
- 配置 `.env`
- 启动后端服务
- 启动前端服务
- 验证数据库连接

### 完成标准

- 后端可以启动并返回健康检查
- 前端可以启动并显示基础页面
- 数据库可以启动
- 后端可以连接数据库
- Python 虚拟环境创建方式写入 README
- 前后端依赖安装方式写入 README
- 数据库启动和配置方式写入 README
- 同事可以从 GitHub 拉取项目后按 README 复现环境
- README 写明前后端启动方式
- `main` 分支保持可运行结构

## 9.2 Module 02：前端页面同步实现

### 目标

项目底座搭好后，两个人先共同实现第一阶段 Web MVP 的主要前端页面和交互壳，让产品形态先清晰起来。该模块先使用 mock 数据和占位接口，不要求后端真实链路全部完成。

这样做的目的不是提前追求页面细节，而是先统一两个人对系统使用流程的理解：

- 用户从哪里进入问答
- 管理员从哪里上传文档
- 回答引用如何展示
- 拒答和追问如何呈现
- 日志和未命中问题如何被查看
- 后续接口需要支撑哪些页面状态

### 包含内容

- 登录页或基础角色入口
- 主布局和导航
- 智能问答页
- 历史会话侧边栏或会话入口
- 文档管理页
- 文档上传组件
- 文档列表和状态展示
- 基础日志页
- 未命中问题页
- 通用 loading / empty / error 状态
- mock API 数据
- 前端类型草案
- 页面级路由

### 不包含内容

- 不要求真实 RAG 回答
- 不要求真实文档解析
- 不要求真实登录鉴权
- 不要求完整权限系统
- 不要求移动端页面
- 不要求语音输入

### 建议边界

- A 负责问答页、会话侧边栏、回答引用展示、追问和拒答状态
- B 负责文档管理页、上传组件、日志页、未命中问题页
- 共同确认主布局和页面路由
- 前端 mock 数据由一个人主改，另一个人 Review
- 前端类型定义由一个人主改，另一个人检查字段是否满足页面需求

如果两个人都主要做前端，可以按页面边界划分：

```text
A：问答相关页面与组件
B：管理相关页面与组件
共享：布局、路由、类型、mock 数据
```

共享文件建议指定主改人：

```text
frontend/src/router/*
frontend/src/types/*
frontend/src/mock/*
frontend/src/layouts/*
frontend/src/styles/*
```

### 建议使用的 skill

该模块建议使用 `frontend-design`。

使用方式：

- 先确定页面信息架构和使用流程
- 再让 Codex 按现有技术栈生成页面骨架
- 对问答页、文档管理页、日志页分别做界面实现
- 实现后用浏览器或截图检查布局、文字溢出、状态展示

注意：

- 本项目是运维工具，不是营销落地页
- 页面应偏工作台风格，重点是清晰、稳定、可扫描
- 不要加入与第一阶段无关的炫技交互
- 不要为了视觉效果牺牲字段完整性和可用性

### 完成标准

- 前端可以启动
- 登录或角色入口可进入系统
- 问答页具备输入框、对话区、引用区、追问/拒答展示
- 文档管理页具备上传入口、文档列表、状态展示
- 日志页和未命中问题页有基础列表
- 页面可以使用 mock 数据完整演示一遍
- 后续 API 契约可以根据页面字段反推和校准

## 9.3 Module 03：API 契约与数据模型

### 目标

先统一接口、状态枚举、核心表结构和返回格式，减少后续联调混乱。

### 包含内容

- 核心状态枚举
- 核心表草案
- QA 返回结构
- 列表接口分页规则
- 错误响应格式
- 文档上传接口契约
- QA 提问接口契约
- 会话、日志、未命中问题接口契约

### 关键状态枚举

```text
kb_document.status:
uploaded / processing / ready / failed / disabled

parse_task.status:
pending / running / success / failed

qa_record.answer_type:
faq / rag / fallback / none

qa_unanswered.status:
new / reviewed / resolved
```

### 建议边界

- A 负责数据库表结构和后端 schema 草案
- B 负责前端类型定义和页面所需字段检查
- `docs/api-contract.md` 或相关接口文档指定一个人主改
- 双方共同 Review 字段命名

### 完成标准

- API 契约文档可供前后端共同开发
- QA 接口返回结构明确
- 文档状态和任务状态明确
- 前端类型和后端 schema 没有明显矛盾

---

## 9.4 Module 04：文档上传与管理

### 目标

跑通“管理员上传文档，系统记录文档状态，前端可以看到文档列表”的闭环。

### 包含内容

- 文档上传接口
- 文件 SHA256 去重
- `kb_document` 记录创建
- `parse_task` 记录创建
- 文档列表接口
- 文档详情接口
- 启用 / 禁用接口
- 前端文档上传组件
- 文档列表页面
- 状态展示
- 失败原因展示占位

### 不包含内容

- 不要求完成真实向量化
- 不要求完成真实问答
- 不要求支持复杂扫描 PDF

### 建议边界

- A 负责上传接口、文档表、任务表、后端状态流转
- B 负责文档管理页、上传组件、列表和状态展示
- API 类型文件由 B 主改，但必须按接口文档
- 数据库迁移文件由 A 主改

### 完成标准

- 可以上传 TXT / MD / DOCX / 文本型 PDF 文件
- 重复文件可以识别或给出明确提示
- 上传后文档列表出现记录
- 页面可以看到状态
- 后端可以返回失败原因字段，即使暂时为空

---

## 9.5 Module 05：文档解析、切块与入库

### 目标

将上传后的文档转换为可检索片段，并保存到数据库。

### 包含内容

- TXT 解析
- Markdown 解析
- DOCX 解析
- 异常 DOCX 兜底解析
- 文本型 PDF 解析
- 文本清洗
- 语义切块
- overlap 处理
- `kb_document_segment` 写入
- 文档状态从 processing 到 ready / failed

### 不包含内容

- 不做复杂 OCR
- 不做图纸识别
- 不做图片表格精细解析

### 建议边界

- A 负责解析器、切块器、入库服务
- B 负责前端解析状态刷新、失败原因展示、文档详情片段预览
- 解析参数文档由 A 主改
- 前端展示字段由 B 主改

### 完成标准

- 样本文档可以解析出文本
- 文档可以切成多个片段
- 片段保留文档名、章节、页码或顺序信息
- 解析失败时有明确 error_message
- 前端能看到 ready / failed 状态

---

## 9.6 Module 06：检索与问答主链路

### 目标

跑通用户提问后，系统可以检索相关片段，生成回答，并展示来源引用。

### 包含内容

- `POST /api/qa/ask`
- Query 基础规范化
- 向量检索
- 关键词检索
- 融合排序
- Top-K 片段返回
- 回答生成
- 引用组装
- 拒答逻辑
- 追问逻辑的第一版
- 问答页联调
- 引用来源展示

### 不包含内容

- 不做实时数据查询
- 不做外部 API 调用
- 不做自动根因分析
- 不做复杂多轮 Agent

### 建议边界

- A 负责检索服务、QA 服务、回答类型、引用结构
- B 负责问答页、消息状态、引用展示、拒答提示
- QA 返回结构由 A 主改，B Review 页面需求
- 前端问答组件由 B 主改，A Review 字段是否完整

### 完成标准

- 用户可以在 Web 页面输入问题
- 后端可以返回 `answer`
- 有效回答必须返回 `references`
- 前端可以展示来源文档和片段摘要
- 无证据时返回 `fallback` 或 `none`
- 每次问答有 `trace_id`

## 9.7 Module 07：会话、日志与未命中问题

### 目标

让问答过程可追踪、可回看、可用于后续优化。

### 包含内容

- `qa_session`
- `qa_record`
- `qa_reference`
- `qa_unanswered`
- 会话列表接口
- 会话详情接口
- 问答日志接口
- 未命中问题接口
- 前端历史会话入口
- 基础日志页
- 未命中问题页

### 建议边界

- A 负责日志、会话、未命中记录的数据写入和查询接口
- B 负责历史会话、日志页、未命中问题页
- 日志字段由 A 主改
- 页面筛选和列表字段由 B 主改

### 完成标准

- 用户提问后产生问答记录
- 有效回答产生引用记录
- 拒答或无证据问题进入未命中记录
- 管理端可以查看日志

## 10. 模块完成定义

一个模块不能只看“代码写完”，必须满足以下 Definition of Done。

```text
1. 模块 Issue 中的完成标准已勾选
2. 前后端或相关链路已至少联调一次
3. 关键接口文档已更新
4. 共享文件变更已 Review
5. 本地验证步骤已写入 PR
6. 已知问题已记录
7. 模块 PR 已合并到 main
```

如果模块没有完全完成，但需要临时合并，必须在 PR 中明确写出：

```text
未完成内容：
影响范围：
后续 Issue：
是否阻塞下一模块：
```

## 12. Codex 使用规范

Codex 可以提高局部开发效率，但不替代两个人的架构判断和范围决策。

## 12.1 适合交给 Codex 的任务

- 生成模块骨架
- 根据接口文档生成 DTO / schema
- 补 repository / service 样板
- 写单元测试
- 检查前后端字段是否一致
- 总结已有代码逻辑
- 生成 PR 描述初稿
- 根据错误日志定位问题
- 整理 Markdown 文档

## 12.2 不建议直接交给 Codex 的任务

- 项目范围拍板
- 数据库最终规范拍板
- 大范围重写
- 未确认边界的跨模块改动
- 未经 Review 的核心检索策略改动

## 12.3 使用 Codex 前的输入格式

向 Codex 提任务时，建议给出：

```text
当前模块：
目标：
允许修改的目录：
禁止修改的目录：
相关接口文档：
验收标准：
```

示例：

```text
当前模块：module-03 文档上传与管理
目标：实现后端文档上传接口
允许修改：backend/app/api/documents.py, backend/app/services/document_service.py
禁止修改：frontend/, docs/api-contract.md
验收标准：上传文件后返回 document_id、status、task_id
```
