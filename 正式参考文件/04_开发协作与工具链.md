# 光伏智能问答系统开发协作与工具链

## 1. 协作原则
- 先写清边界，再开始编码
- 先落文档，再落代码
- 先做可验证闭环，再做扩展能力
- 用 Codex 提升局部开发效率，不把架构决策交给工具

## 2. Codex 的使用方式
### 适合交给 Codex 的任务
- 补模块骨架
- 写 DTO / schema / repository
- 写单元测试
- 做局部重构
- 检查字段命名与接口一致性
- 生成重复样板代码
- 帮忙读文件并总结模块逻辑

### 不建议直接交给 Codex 的任务
- 核心架构拍板
- 产品范围定义
- 数据库最终规范决策
- 不明确边界的大范围重写

## 3. 推荐 skill 使用方式
- `brainstorming`：讨论架构和技术路线时使用
- `planning-with-files`：复杂任务和长期任务时使用
- `frontend-design`：前端页面实现时使用
- `markitdown`：文档转换和内容提取时使用

## 4. 两人分工建议
### 开发者 A
- core-api
- 检索链路
- 问答编排
- 引用组装
- 日志追踪

### 开发者 B
- ingest-worker
- 解析器
- 切块器
- embedding
- 入库任务状态

### 双方共同负责
- 数据库 schema
- API 契约
- 验收样例
- PR review
- 问题定位

## 5. 推荐工作流
1. 先写文档和接口约定
2. 再让 Codex 生成模块骨架
3. 人工检查边界和异常处理
4. 逐个模块实现
5. 小步联调
6. 记录问题并更新 findings / progress

## 6. 工具链建议
- Git 用于协作与版本控制
- Codex 用于代码生成与局部审阅
- planning-with-files 用于任务持久化
- 代码风格和接口规范用文档约束
- 后续如有需要，再补测试和可视化工具

## 7. 结论
工具的目标是加速实现，不是替代决策。对你们现在这个阶段，最重要的是保持边界清晰、任务可追踪、输出可验证。
