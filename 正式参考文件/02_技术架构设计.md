# 光伏智能问答系统第一阶段技术架构设计

## 1. 架构目标
第一阶段采用“逻辑服务化，物理双服务”的方案，在保持边界清晰的同时控制实现复杂度。

## 2. 服务拆分
### core-api
职责：
- 对外 HTTP API
- 问答入口
- 会话管理
- 检索编排
- FAQ 优先匹配
- 回答生成
- 引用组装
- 拒答与兜底
- 日志与追踪

### ingest-worker
职责：
- 文档任务处理
- 文档解析
- 文本清洗
- 语义切块
- embedding 生成
- 写入片段和向量
- 更新任务状态与错误信息

## 3. 总体数据流
上传文件 -> core-api 创建 parse_task -> ingest-worker 处理任务 -> 文档片段入库 -> core-api 面向用户检索与回答。

## 4. 关键模块划分
### core-api 内部模块
- auth
- documents
- retrieval
- qa
- sessions
- logs
- models
- llm adapter

### ingest-worker 内部模块
- task runner
- parsers
- cleaners
- chunkers
- embedder
- persistence

## 5. 技术选型
- 前端：Vue 3 + TypeScript + Vite + Element Plus + Pinia
- 后端：FastAPI + SQLAlchemy 2.0 + Pydantic
- 数据库：PostgreSQL 16 + pgvector
- 文档解析：TXT / MD 直接读取，DOCX 优先使用 python-docx，PDF 使用 PyMuPDF
- 检索：关键词检索 + 向量检索 + 融合排序
- 模型适配：embedding provider + chat provider 分层封装

### 5.2 检索链路详细设计
第一阶段检索链路采用“关键词检索 + 向量检索 + 融合排序”的混合方案，目标是在光伏运维场景下同时兼顾术语精确命中和语义召回能力。

#### 5.2.1 向量模型建议
- 第一阶段向量模型固定为 `BAAI/bge-m3`
- 第一阶段仅使用其 dense embedding 能力，不启用 sparse 或 multi-vector 能力
- 首选要求：中文效果稳定、支持技术文档、维度明确、推理成本可控
- 工程约束：
  - 向量维度必须固定，并由 `embedding_provider.get_dimension()` 返回
  - 文档切块和查询问题必须使用同一模型生成向量
  - 模型切换时必须触发重建向量索引

选型原因：
- 中文检索能力较强
- 适合技术文档和问答场景
- 兼顾后续扩展空间
- 第一阶段只用 dense 模式即可满足需求

#### 5.2.2 向量相似度匹配
- pgvector 固定使用余弦距离
- 向量索引固定使用 HNSW
- 文档片段写入后建立向量索引
- 在线查询时对用户问题向量化，返回 Top-K 候选片段

适用原因：
- 运维问答中存在大量自然语言提问
- 同义表达、口语化表达较多
- 需要弥补纯关键词检索在语义理解上的不足

#### 5.2.3 关键词检索建议
第一阶段关键词检索固定采用 `jieba + PostgreSQL 全文检索`，并对型号、编号、故障码等短字符串检索补充 `pg_trgm` 能力。

重点覆盖以下内容：
- 设备型号
- 逆变器编号
- 故障码
- 章节标题
- 专业术语
- 数值和参数名称

适用原因：
- 光伏场景中很多查询依赖精确术语匹配
- 型号、故障码、参数类问题对关键词检索更敏感
- 纯向量检索容易在精确匹配问题上发生语义漂移

工程约束：
- 入库时先对中文文本做分词
- 维护一份光伏领域分词词典
- 分词结果写入可检索字段
- 对短字符串模糊匹配使用 `pg_trgm`

#### 5.2.4 融合排序建议
第一阶段融合排序固定采用 `RRF（Reciprocal Rank Fusion）`：
- 向量检索负责语义召回
- 关键词检索负责精确召回
- 融合排序输出最终 Top-K

推荐顺序：
1. 关键词检索返回 Top-N
2. 向量检索返回 Top-N
3. 对结果去重
4. 进行融合排序
5. 选出最终用于回答生成的证据片段

默认建议：
- 关键词检索 Top 20
- 向量检索 Top 20
- 融合后返回 Top 5 作为回答证据

#### 5.2.5 第一阶段调参重点
- Top-K 数量
- 分词词典质量
- 最低证据阈值
- FAQ 优先命中阈值
- 拒答阈值

第一阶段验收前需要基于真实文档和 golden QA 集做至少一轮调参，而不是只凭经验设置默认值。

### 5.1 模型适配层最小接口约定
Embedding Provider:

```python
async def embed(texts: list[str]) -> list[list[float]]
def get_dimension() -> int
```

Chat Provider:

```python
async def chat(messages: list[dict], **kwargs) -> ChatResponse
```

`ChatResponse` 至少包含以下字段：
- `content`
- `finish_reason`
- `usage`

## 6. 任务通信方式
第一阶段建议优先使用数据库任务表轮询，避免过早引入重型消息队列。后续如有吞吐压力，再引入 Redis 或任务队列组件。

### 6.1 任务领取约束
- 任务领取必须使用原子操作，避免多个 worker 实例争抢同一任务。
- 推荐使用 `UPDATE ... SET status='running' WHERE id=? AND status='pending' RETURNING *` 或 `SELECT ... FOR UPDATE SKIP LOCKED` 领取任务。
- 默认轮询间隔建议为 5 秒。
- 状态流转必须具备幂等性，确保重复执行不会造成重复入库。

## 7. 设计原则
- 上传链路与问答链路解耦
- core-api 不直接执行重解析和向量化
- 所有模型调用都通过适配层
- 问答接口不写死为 RAG 专属命名
- 模块边界为第二阶段扩展保留空间

## 8. 结论
该架构适合两人团队的第一阶段交付，既能保持工程边界，又不会让部署和联调复杂度过高。
