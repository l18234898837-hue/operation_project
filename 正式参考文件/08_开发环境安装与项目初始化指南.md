## 4. Python 安装与检查

## 4.1 推荐版本

推荐安装：

```text
Python 3.11.x
```

建议优先使用 `3.11`，原因是：

- 生态稳定
- 常见依赖兼容性更好
- 对文档处理、FastAPI、SQLAlchemy、PyMuPDF 等支持成熟

`3.12` 也可以，但如果你们希望减少兼容性风险，优先选 `3.11`。

## 4.2 安装步骤

1. 打开 Python 官方下载页面。
2. 下载 Windows 64-bit 安装包。
3. 运行安装程序。
4. 安装时务必勾选：

```text
Add python.exe to PATH
```

5. 选择标准安装即可。

## 4.3 安装后检查

打开 PowerShell，执行：

```powershell
python --version
```

预期结果示例：

```text
Python 3.11.9
```

继续检查：

```powershell
pip --version
```

预期结果示例：

```text
pip 24.x from ... (python 3.11)
```

如果 `python` 命令找不到：

- 先关闭当前终端重新打开
- 再试一次
- 如果仍失败，检查 PATH
- 或重新安装并确认勾选 `Add python.exe to PATH`

## 4.4 建议再检查

```powershell
python -m pip install --upgrade pip
```

这一步建议执行，用于升级 `pip`。

## 5. venv 使用方式

## 5.1 为什么必须使用 venv

本项目后端依赖较多，必须使用虚拟环境，避免：

- 全局 Python 依赖冲突
- 不同项目之间互相污染
- 协作者环境不一致

## 5.2 创建虚拟环境

假设项目根目录为：

```text
D:\桌面\文件\operation_project
```

建议虚拟环境放在：

```text
backend\.venv
```

进入项目根目录后执行：

```powershell
cd D:\桌面\文件\operation_project
python -m venv backend\.venv
```

## 5.3 激活虚拟环境

PowerShell 中执行：

```powershell
.\backend\.venv\Scripts\Activate.ps1
```

激活成功后，终端前面通常会出现：

```text
(.venv)
```

## 5.4 如果 PowerShell 拒绝执行脚本

如果出现执行策略问题，可以先执行：

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

然后重新打开 PowerShell，再激活虚拟环境。

## 5.5 检查当前 Python 是否来自虚拟环境

激活后执行：

```powershell
where python
```

预期结果中应包含：

```text
...\backend\.venv\Scripts\python.exe
```

如果没有，说明虚拟环境没有正确激活。

## 7. PostgreSQL 16 安装与检查

## 7.1 推荐版本

推荐安装：

```text
PostgreSQL 16
https://www.enterprisedb.com/downloads/postgres-postgresql-downloads
```

## 7.2 安装步骤

1. 下载 PostgreSQL 16 Windows 安装包。
2. 运行安装程序。
3. 安装时记录以下信息：

```text
安装目录
superuser 用户名
superuser 密码
端口
```

建议默认：

```text

端口：5432
```

4. 安装 pgAdmin 可选，但建议安装一个数据库可视化工具。

## 7.3 安装后检查

检查服务是否存在：

```powershell
Get-Service | Where-Object { $_.Name -like '*postgres*' }
```

预期结果：可以看到 PostgreSQL 服务处于 `Running` 或至少存在。

检查客户端命令：

```powershell
psql --version
```

预期结果示例：

```text
psql (PostgreSQL) 16.x
```

如果 `psql` 不存在，通常是 PostgreSQL 的 `bin` 目录没有加到 PATH。可以：

- 手动将 PostgreSQL 安装目录下的 `bin` 加到 PATH
- 或直接使用完整路径调用

## 7.4 登录检查

执行：

```powershell
psql -U postgres -h localhost -p 5432
```

输入密码后，如果能进入 PostgreSQL 命令行，说明数据库本体安装成功。

进入后执行：

```sql
SELECT version();
```

如果能返回 PostgreSQL 版本信息，则数据库可用。

## 8.1 推荐安装流程（Windows + PostgreSQL 16）

## 8.1.1 准备官方下载链接

## 8.1.2 确认 PostgreSQL 16 已安装

如果还没装 PostgreSQL 16，按下面步骤安装：

1. 打开 PostgreSQL Windows 官方下载页：
   [https://www.postgresql.org/download/windows/](https://www.postgresql.org/download/windows/)

3. 下载 PostgreSQL 16 的 Windows 64 位安装器。

8. 安装完成后，在 PowerShell 中检查：

   ```powershell
   psql --version
   ```

   预期看到：


如果已经安装 PostgreSQL 16，只需要确认真实安装目录即可。

## 8.1.3 安装 Git

如果还没装 Git：

1. 打开 Git Windows 下载页：

后续命令都建议在：

```text
x64 Native Tools Command Prompt for VS 2022
```

中执行。

不要在普通 PowerShell 中编译。

在该终端中执行：

```cmd

```

如果能看到 Microsoft C/C++ 编译器信息，说明 `cl` 可用。

继续执行：

```cmd
nmake /?
```

如果能看到 `nmake` 帮助信息，说明构建工具可用。

## 8.1.6 设置 PostgreSQL 安装目录 PGROOT

在 `x64 Native Tools Command Prompt for VS 2022` 中执行：

```cmd
set "PGROOT=C:\Program Files\PostgreSQL\16"
```

如果你的 PostgreSQL 安装在其他位置，请改成真实路径。

检查 `PGROOT` 是否正确：

```cmd
dir "%PGROOT%"
```

应该能看到：

```text
bin
include
lib
share
```

如果看不到这些目录，说明 `PGROOT` 设置错了，先不要继续。

## 8.1.7 下载 pgvector 源码

推荐使用 Git 下载：

```cmd
cd %TEMP%
git clone --branch v0.8.2 https://github.com/pgvector/pgvector.git
cd pgvector
```

如果不想使用 Git，也可以：

1. 打开 [https://github.com/pgvector/pgvector](https://github.com/pgvector/pgvector)
2. 点击 `Code`
3. 点击 `Download ZIP`
4. 解压到：

   ```text
   C:\temp\pgvector
   ```
5. 在 `x64 Native Tools Command Prompt for VS 2022` 中进入目录：

   ```cmd
   cd /d C:\temp\pgvector
   ```

## 8.1.8 编译 pgvector

在 `pgvector` 源码目录中执行：

```cmd
nmake /F Makefile.win
```

如果命令执行完成且没有报错，说明编译成功。

## 8.1.9 安装 pgvector 到 PostgreSQL 16

继续执行：

```cmd
nmake /F Makefile.win install
```

这一步会把扩展文件复制到 PostgreSQL 16 的目录中，通常包括：

```text
%PGROOT%\share\extension
%PGROOT%\lib
```

如果安装时提示权限问题，关闭终端，重新以管理员身份打开 `x64 Native Tools Command Prompt for VS 2022`，重新设置 `PGROOT`，再执行安装命令。

## 8.2 验证安装是否成功

## 8.2.1 检查扩展文件

检查控制文件目录：

```text
C:\Program Files\PostgreSQL\16\share\extension
```

应该能看到：

```text
vector.control
vector--*.sql
```

检查动态库目录：

```text
C:\Program Files\PostgreSQL\16\lib
```

应该能看到：

```text
vector.dll
```

如果这些文件不存在，说明 `pgvector` 还没有真正安装到 PostgreSQL 16。

## 8.2.2 创建或连接项目数据库

如果还没有项目数据库，执行：

```powershell
psql -U postgres -h localhost -p 5432
```

进入 PostgreSQL 后执行：

```sql
CREATE DATABASE pv_qa;
\c pv_qa
```

如果已经有数据库，直接连接：

```powershell
psql -U postgres -h localhost -p 5432 -d pv_qa
```

## 8.2.3 启用 vector 扩展

在 `pv_qa` 数据库中执行：

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

如果这一步成功，说明 PostgreSQL 已经能识别 `pgvector` 扩展。

## 8.2.4 检查扩展列表

执行：

```sql
\dx
```

结果中应能看到：

```text
vector
```

也可以执行：

```sql
SELECT extname FROM pg_extension WHERE extname = 'vector';
```

如果返回 `vector`，说明扩展启用成功。

## 8.2.5 创建测试表验证 vector 类型

执行：

```sql
CREATE TABLE IF NOT EXISTS vector_test (
    id bigserial PRIMARY KEY,
    embedding vector(3)
);
```

插入测试数据：

```sql
INSERT INTO vector_test (embedding) VALUES ('[1,2,3]');
```

查询：

```sql
SELECT * FROM vector_test;
```

如果能正常插入和查询，说明 `pgvector` 已经可以使用。

## 8.2.6 安装成功的最终标准

满足下面这些条件，就可以认为安装成功：

```text
1. PostgreSQL 16 正常运行
2. %PGROOT%\share\extension 中存在 vector.control
3. %PGROOT%\share\extension 中存在 vector--*.sql
4. %PGROOT%\lib 中存在 vector.dll
5. psql 可以连接到 pv_qa 数据库
6. CREATE EXTENSION IF NOT EXISTS vector; 执行成功
7. \dx 能看到 vector
8. 可以创建 vector(n) 字段
9. 可以插入并查询向量测试数据
```

## 8.3 常见问题与排查

## 8.3.1 `CREATE EXTENSION vector;` 为什么会报错

`CREATE EXTENSION vector;` 只是启用扩展，不是安装扩展。

也就是说，必须先把下面这些文件安装到 PostgreSQL 目录：

```text
vector.control
vector--*.sql
vector.dll
```

然后数据库里才能执行：

```sql
CREATE EXTENSION vector;
```

## 8.3.2 报错：extension "vector" is not available

含义：

- PostgreSQL 找不到 `vector.control`

排查：

1. 检查 `%PGROOT%\share\extension\vector.control` 是否存在
2. 检查你是否安装到了 PostgreSQL 16 的目录
3. 检查 `PGROOT` 是否设置正确
4. 重新执行 `nmake /F Makefile.win install`

## 8.3.3 报错：could not open extension control file

含义：

- 控制文件不存在或路径不对

排查：

1. 打开 `C:\Program Files\PostgreSQL\16\share\extension`
2. 查看是否有 `vector.control`
3. 如果没有，回到源码目录重新执行：

   ```cmd
   nmake /F Makefile.win install
   ```

## 8.3.4 报错：permission denied to create extension

含义：

- 当前数据库用户权限不够

处理：

- 使用 `postgres` 超级用户执行
- 或让具备权限的管理员执行

## 8.3.5 编译时报错：找不到 cl

含义：

- 没有在 `x64 Native Tools Command Prompt for VS 2022` 中执行
- 或没有安装 C++ 工具链

处理：

1. 打开 Visual Studio Installer
2. 勾选 `使用 C++ 的桌面开发`
3. 确认勾选 `MSVC 生成工具`
4. 重新打开 `x64 Native Tools Command Prompt for VS 2022`
5. 执行：

   ```cmd
   cl
   ```

## 8.3.6 编译时报错：找不到 nmake

含义：

- C++ Build Tools 没装好
- 或打开了普通终端

处理：

1. 确认 Visual Studio Build Tools 已安装
2. 确认 `使用 C++ 的桌面开发` 工作负载已安装
3. 打开 `x64 Native Tools Command Prompt for VS 2022`
4. 执行：

   ```cmd
   nmake /?
   ```

## 8.3.7 `nmake install` 报权限问题

含义：

- 当前命令行没有权限写入 PostgreSQL 安装目录

处理：

1. 关闭当前终端
2. 以管理员身份打开 `x64 Native Tools Command Prompt for VS 2022`
3. 重新设置 `PGROOT`
4. 回到 `pgvector` 源码目录
5. 重新执行：

   ```cmd
   nmake /F Makefile.win install
   ```

## 8.3.8 安装成功了，但数据库还是启用失败

可能原因：

- 连接的 PostgreSQL 实例不是你安装扩展的那个实例
- 电脑上装了多个 PostgreSQL 版本
- `PGROOT` 指向了错误版本

检查：

```sql
SHOW data_directory;
SHOW config_file;
```

同时确认你安装扩展时使用的目录是：

```text
C:\Program Files\PostgreSQL\16
```

## 8.3.9 不想本机编译怎么办

如果觉得 Windows 本机编译太麻烦，可以改成 Docker 方案：

- 安装 Docker Desktop
- 使用带 `pgvector` 的 PostgreSQL 镜像
- 在 `docker-compose.yml` 中统一数据库环境

这种方式对团队协作更统一，但需要提前安装 Docker Desktop。

当前如果你已经直装了 PostgreSQL 16，就先按本节本机编译方案处理。

## 8.4 补充说明：pgvector 的作用

`pgvector` 是第一阶段向量检索的核心扩展，没有它就无法在 PostgreSQL 中做 embedding 存储和向量相似度查询。

## 9. Redis 安装与检查

## 9.1 当前阶段为什么先装 Redis

虽然你们当前文档里第一阶段的任务处理建议优先使用数据库任务表轮询，但先把 Redis 装好有几个好处：

- 后续如果要引入缓存更方便
- 后续如果要接任务队列更方便
- 不影响当前架构，但能提前统一环境

当前阶段可以先装好、先验证，不一定马上深度使用。

## 9.2 Windows 下的建议

Windows 下 Redis 原生支持不如 Linux 完整，常见选择有两种：

### 方案 A：使用 Docker 运行 Redis

优点：

- 最省心
- 与团队环境更容易统一

缺点：

- 需要 Docker Desktop

### 方案 B：使用 Windows 可运行的 Redis 发行版

优点：

- 不依赖 Docker

缺点：

- 来源和维护情况不如 Docker 方案稳定

如果你们当前还不打算立即装 Docker，那么可以先记录 Redis 为“本机可选组件”，等真正用到时再统一容器化。

如果你们希望现在就把 Redis 配齐，且不排斥 Docker，我会推荐 Redis 使用 Docker 方式。

## 9.3 检查方式

如果 Redis 已安装并运行，执行：

```powershell

```

预期结果：

```text
PONG
```

如果没有 `redis-cli`，说明 Redis 客户端没有安装或 PATH 没配好。
