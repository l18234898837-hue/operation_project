# 光伏智能问答系统数据库与接口设计

## 1. 设计原则
- 主键统一使用 UUID
- 状态字段统一使用枚举
- 核心业务字段显式存储
- JSON 仅用于扩展信息
- 所有关键表保留 created_at 和 updated_at
- 所有异步任务保留 status、error_message、finished_at
- 所有问答结果必须可追溯到 trace_id
- 所有回答都必须可追溯到引用来源
- 所有文档片段都要保留原文和清洗后文本

## 2. 核心状态枚举
### kb_document.status
- uploaded
- processing
- ready
- failed
- disabled

### parse_task.status
- pending
- running
- success
- failed

### qa_record.answer_type
- faq
- rag
- fallback
- none

### qa_unanswered.status
- new
- reviewed
- resolved

## 3. 核心表
### kb_document
文档主表，记录文件元数据、状态、哈希、路径、启用禁用等信息。

### parse_task
入库任务表，记录解析流程、重试次数、错误信息、执行耗时。

### kb_document_segment
文档片段表，记录切块后的正文、清洗文本、页码、章节、embedding、排序信息。

### faq_item
FAQ 表，供高优先级直答匹配。

### qa_session
会话主表，承载前端历史会话。

### qa_record
问答记录表，保存每次提问、回答、置信度、决策信息、耗时、模型信息。

### qa_reference
回答引用表，保存每条回答关联的证据片段。

### qa_unanswered
未命中问题表，记录拒答问题及后续处理状态。

## 4. 建议接口
### documents
- POST /api/documents/upload
- GET /api/documents/list
- GET /api/documents/{id}
- POST /api/documents/{id}/enable
- POST /api/documents/{id}/disable

### tasks
- GET /api/tasks/{id}

### qa
- POST /api/qa/ask

### sessions
- GET /api/sessions
- GET /api/sessions/{id}

### logs
- GET /api/logs/qa

### unanswered
- GET /api/unanswered
- PATCH /api/unanswered/{id}

### 4.1 列表接口标准参数
对于列表型接口，建议统一支持以下查询参数：

```text
?page=1&page_size=20&status=ready&sort=created_at:desc
```

可选筛选项按资源类型补充，例如：
- `status`
- `enabled`
- `answer_type`
- `date_from`
- `date_to`

### 4.2 统一错误响应格式

```json
{
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "文档不存在或已被删除"
  }
}
```

## 5. QA 返回建议结构
```json
{
  "answer": "...",
  "answer_type": "faq|rag|fallback|none",
  "confidence": 0.86,
  "references": [],
  "trace_id": "...",
  "fallback_reason": null
}
```

## 6. 索引建议
- `kb_document.file_sha256` 唯一索引
- `kb_document.status` 普通索引
- `parse_task.document_id` 索引
- `parse_task.status` 索引
- `kb_document_segment.document_id` 索引
- `kb_document_segment.embedding` 向量索引
- `qa_record.session_id` 索引
- `qa_record.trace_id` 唯一或普通索引
- `qa_reference.qa_record_id` 索引

## 6.1 关键关系字段约束
### qa_reference
`qa_reference` 至少应包含以下核心字段：
- `qa_record_id`：关联问答记录
- `segment_id`：关联 `kb_document_segment`
- `relevance_score`：该片段相关性分数
- `excerpt`：引用摘录，便于前端直接展示

如果需要支持更通用的第二阶段引用结构，可在此基础上继续保留：
- `ref_type`
- `document_id`
- `page_no`
- `section_title`
- `ref_metadata_json`

## 7. 结论
这套数据库设计重点是让文档处理、问答过程和引用链路都具备可追踪、可调试、可扩展的能力。
