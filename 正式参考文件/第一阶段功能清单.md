# 光伏智能问答系统第一阶段功能清单

> 版本：v1.0  
> 适用阶段：第一阶段（文档知识库 RAG）  
> 适用对象：产品、开发、测试、项目协作人员

---

## 1. 文档目的

本文档用于在第一阶段产品边界已明确的前提下，进一步整理第一阶段的功能清单，帮助团队完成以下目标：

- 明确第一阶段必须交付的功能范围
- 区分核心必做、增强可选和明确不做项
- 为两人协作分工、排期和验收提供统一依据

---

## 2. 第一阶段目标概述

第一阶段目标不是做一个全能型智能助手，而是完成一个面向光伏运维/技术人员的 Web MVP，优先跑通以下闭环：

1. 文档可以上传并入库
2. 用户可以发起自然语言提问
3. 系统可以基于文档和案例库检索证据
4. 系统可以输出带引用的回答
5. 信息不足时可以追问
6. 无依据或超范围时可以拒答

---

## 3. 第一阶段功能分级

为避免范围失控，第一阶段功能分为三类：

- `P0 必做`：没有这些功能，第一阶段闭环无法成立
- `P1 建议做`：对可用性和演示效果有明显帮助，但可以在闭环跑通后补齐
- `Out`：明确不纳入第一阶段

---

## 4. P0 必做功能清单

## 4.1 文档知识库管理

### 4.1.1 文档上传

- 支持上传第一阶段约定范围内的文档
- 第一阶段优先支持：
  - TXT
  - Markdown
  - DOCX
  - 文本型 PDF

目标：

- 管理员能够手动导入样本文档和故障案例文档

### 4.1.2 文档去重

- 对上传文档进行唯一性识别
- 避免同一文档重复入库

目标：

- 保持知识库内容整洁，减少重复片段干扰检索

### 4.1.3 文档解析

- 根据文件类型选择解析方式
- 将原始文档转换为可检索文本
- 对异常 DOCX、结构不稳定文档保留兜底解析能力

目标：

- 样本文档可稳定解析为结构化文本内容

### 4.1.4 文本清洗与切块

- 清洗无意义空白、异常换行、噪音符号
- 按标题、段落、语义边界优先切块
- 保留适度 overlap，避免上下文断裂

目标：

- 形成适合召回的知识片段

### 4.1.5 向量化与入库

- 对切块内容进行向量化
- 写入向量库和业务数据库
- 保存文档、片段、来源信息等基础数据

目标：

- 问答链路可以基于这些片段完成检索

### 4.1.6 文档状态管理

- 展示文档解析状态
- 支持启用/禁用文档
- 能查看解析失败原因

目标：

- 管理员能知道哪些文档可用，哪些需要处理

---

## 4.2 问答主链路

### 4.2.1 用户提问入口

- 提供文字输入框
- 支持发送问题
- 支持新建对话

目标：

- 用户可以从 Web 页面发起问答

### 4.2.2 Query 预处理

- 对用户问题做基础规范化
- 可参考 Query 改写策略，对口语化、模糊化问题做轻量改写或规范化
- 对明显模糊但相关的问题触发追问策略
- 对无关问题或超范围问题做分类处理

目标：

- 提高检索稳定性，避免一上来就乱答

说明：

- Query 改写在第一阶段只建议做轻量化策略
- 目标是提升检索表达适配性，而不是引入复杂多路推理

### 4.2.3 文档检索

- 基于知识片段完成召回
- 第一阶段建议采用混合检索：
  - 向量检索
  - 关键词检索
  - 融合排序

目标：

- 能召回与问题最相关的文档片段

### 4.2.4 回答生成

- 基于检索证据生成回答
- 严格遵守产品边界说明中的回答深度约束
- 统一采用结构化回答方式

目标：

- 输出稳定、清晰、可理解的排障型回答

### 4.2.5 引用展示

- 每条有效回答都必须附带来源引用
- 引用至少应包含：
  - 文档名
  - 片段或章节信息
  - 相关内容摘要

目标：

- 让回答可追溯、可核验

### 4.2.6 追问机制

- 对相关但信息不足的问题先追问
- 最多追问 2 轮
- 两轮后若仍无法补齐信息，按文档边界规则降级或停止回答

目标：

- 减少空泛回答，同时避免无限追问

### 4.2.7 拒答与兜底

- 对无关问题、超范围问题、证据不足问题统一拒答
- 支持固定兜底话术

目标：

- 避免幻觉和错误指导

---

## 4.3 会话与记录能力

### 4.3.1 会话历史

- 保存用户问答记录
- 支持查看历史会话

目标：

- 用户可以回看之前的问答内容

### 4.3.2 问答日志

- 记录问题、回答类型、来源数量、处理结果、时间等信息
- 保留 trace_id 或等价追踪标识

目标：

- 便于排查问题、做效果回溯和后续调优

### 4.3.3 未命中问题记录

- 记录没有成功回答或证据不足的问题
- 支持管理员查看

目标：

- 为后续知识库补充和 FAQ 建设提供依据

---

## 4.4 Web MVP 页面

### 4.4.1 智能问答页

至少包含：

- 输入框
- 发送按钮
- 对话区
- 回答区
- 引用来源区
- 加载中状态
- 无结果/拒答提示

### 4.4.2 文档管理页

至少包含：

- 文档上传入口
- 文档列表
- 解析状态展示
- 启用/禁用操作
- 失败原因查看

### 4.4.3 基础日志页

至少包含：

- 问题
- 回答类型
- 时间
- 来源片段数量
- trace_id

## 5. P1 建议做功能清单

以下功能对第一阶段演示效果和后续优化有帮助，建议在 P0 跑通后补齐：

## 5.1 推荐问题

- 首页展示若干推荐提问
- 帮助用户理解系统擅长回答什么

## 5.2 回答反馈

- 支持点赞/点踩
- 为后续 bad case 分析提供反馈信号

## 5.3 回答复制

- 支持复制回答内容

## 5.4 重新生成

- 在相同证据基础上允许重新生成一次回答

## 5.5 文档详情查看

- 查看文档基础信息、解析结果或片段摘要

## 5.6 简版分类管理

- 按逆变器、组件、线缆、SVG 等类型做简单分类

## 5.7 评测辅助能力

- 构建 golden QA
- 建立基础评测脚本
- 对 Recall、Faithfulness 等指标进行初步验证

## 5.8 历史会话页

- 提供会话列表
- 支持查看会话详情

## 5.9 未命中问题页

- 展示未成功回答的问题
- 支持后续人工整理和知识库补充

---

## 6. 明确不纳入第一阶段的功能

以下内容明确不纳入第一阶段：

- 移动端悬浮助手
- 语音输入与语音识别
- FAQ 优先匹配机制
- 实时数据查询
- 外部数据库/API 接入
- SCADA / DCS 接入
- 工单系统接入
- 主动预警
- 自动根因分析
- 时序建模
- 预测类能力
- 多智能体协作
- 复杂图纸、扫描件、重 OCR 文档治理
- 完整运营后台

---

## 7. 推荐实现顺序

为减少混乱，建议第一阶段功能按以下顺序推进：

1. 文档上传、解析、切块、入库跑通
2. 检索主链路跑通
3. Query 轻量规范化、追问、拒答跑通
4. 回答生成、引用展示跑通
5. Web 问答页和文档管理页联调
6. 基础日志与增强功能补齐

---

## 8. 两人协作的功能拆分建议

若当前由两人协作开发，建议按以下方式拆分：

## 8.1 开发 A：后端 / 知识库 / 检索链路

负责：

- 文档上传与解析
- 文本切块与向量化
- 数据库表设计
- Query 轻量改写与检索服务
- 回答生成
- 引用结构
- 追问与拒答逻辑
- 日志与未命中记录

## 8.2 开发 B：前端 / Web MVP / 联调

负责：

- Web 项目初始化
- 问答页
- 文档管理页
- 日志页
- 页面状态与交互
- 与后端接口联调

## 8.3 共同负责

- API 契约确认
- 样本问题梳理
- 验收测试
- 演示流程准备

---

## 9. 第一阶段验收视角下的最小功能包

如果团队时间紧张，第一阶段至少要保证以下最小功能包成立：

- 文档上传
- 文档解析
- 文本切块与入库
- 用户提问
- 文档检索
- 回答生成
- 来源引用
- 追问
- 拒答
- 基础问答页

只要这套最小功能包成立，第一阶段就具备“可演示、可试用、可继续迭代”的基础。

---

## 10. 一句话总结

第一阶段功能清单的核心不是“功能越多越好”，而是优先交付一个真正能用的文档问答闭环：

**能上传资料、能回答问题、能给出处、答不准不乱答。**
