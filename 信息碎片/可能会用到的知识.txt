### 1. RAG 中为什么要做 Query 改写

#### 概念解释

在 RAG 场景里，用户的提问方式和知识库中的文档表达方式往往并不一致。用户问题可能比较口语化、模糊，或者夹杂很多冗余信息；而知识库里的内容通常更书面化、结构化、专业化。如果直接拿原始问题去检索，容易出现召回不准、匹配不到关键信息、结果覆盖不全等问题。

所以 Query 改写的核心目标，就是把“用户的提问形式”转换成“更适合检索的表达形式”，从而提升召回质量和检索精准度。本质上，它解决的是“用户语言”和“知识库语言”之间的表达鸿沟。

#### 面试表达

如果面试官问我为什么要做 Query 改写，我一般会这样回答：

在 RAG 系统里，用户的提问往往不是为检索设计的，而知识库文档通常是按正式、结构化方式组织的，所以两者之间会存在表达差异。比如用户的问题可能很口语化、很模糊，甚至带有很多无关信息，如果直接去检索，效果通常不会太好。  
因此 Query 改写的核心目的，就是把用户问题转换成更清晰、更规范、更适合检索的形式，让检索结果更精准，召回更稳定。

#### 常见方法

##### 1. 直接查询重写

思路是利用大模型把用户原始问题改写得更清晰、更专业、更结构化。

例如用户问：“某个产品的最新版本咋样？”  
可以改写成：“请介绍某个产品最新版本的功能特点和性能提升。”

这样改写之后，问题表达会更规范，关键词也更明确，更容易匹配到知识库里的相关文档。

适用场景：

- 用户提问口语化明显
- 问题描述模糊，关键信息不突出
- 希望在不改变用户意图的前提下提升检索命中率

##### 2. 多查询生成

思路是围绕一个问题生成多个相似但角度不同的查询，再并行检索，最后合并结果。

例如用户问：“如何提高 RAG 系统的性能？”

可以扩展成多个检索变体：

- RAG 性能优化的方法
- 提高 RAG 检索准确率的技巧
- RAG 系统的性能瓶颈与解决方案

这样做的优势是覆盖面更广，不容易漏掉关键文档，尤其适合用户问题比较笼统、单条查询无法充分表达检索意图的场景。

适用场景：

- 用户问题比较宽泛
- 一个问题可能对应多个知识角度
- 更关注召回覆盖率而不是单次检索的极致精确性

##### 3. 查询分解

这类方法主要针对复杂问题。某些用户提问本身包含多个维度、多层约束，直接检索往往会把不同意图混在一起。

例如用户问：“对比 GPT 和某模型在代码生成和长文本理解上的表现。”

这个问题里同时包含了：

- 模型对比
- 代码生成维度
- 长文本理解维度

更合理的做法是把它拆成多个子问题，分别检索，再把结果整合起来。这样不仅检索更清晰，也更有利于后续答案组织。

适用场景：

- 问题包含多个比较维度
- 一个问题里混合多个子任务
- 需要结构化回答复杂问题

##### 4. 假设文档嵌入

这个方法不是直接改写问题，而是先让模型根据问题生成一段“假设答案”或者“假设文档片段”，再用这段内容去做检索。

它的核心原因是：有时候用户问题和知识库答案在语义表达上差异很大，直接用问题检索未必能命中；但如果先生成一段更接近目标文档风格的假设文本，再用它检索，往往更容易匹配到真正相关的内容。

本质上，这是一种把“问题表示”转成“答案表示”来做召回增强的方法。

适用场景：

- 问题和文档之间表达差异大
- 直接 query-to-document 检索效果不稳定
- 希望增强语义层面的匹配能力

##### 5. 后退提示

这个思路和“让问题更具体”相反，它会先把问题抽象成更宏观、更通用的形式，再去检索背景知识。

例如用户问：“某个产品的价格是多少？”

如果知识库里没有这个产品价格的直接答案，那么直接检索可能失败。这时可以先后退一步，改成更通用的问题，例如：

- 这个公司的定价策略是什么
- 同类产品的价格体系是什么

先检索这些更上层的背景信息，再回到原始问题做回答。这样即使知识库里没有直接答案，也仍然可以给用户一个相对合理、有依据的回复。

适用场景：

- 原始问题过于具体，知识库未必有直接答案
- 更适合先检索背景知识再落回具体问题
- 希望增强系统在“无直接命中”场景下的可回答性

#### 项目里的组合方式

在实际项目里，这些方法通常不是单独使用，而是组合使用。

比较常见的做法是：

1. 先做查询重写，把原始问题规范化。
2. 再做多查询生成，扩展不同检索角度。
3. 对复杂问题再做查询分解，避免多意图混杂。
4. 检索完成后结合重排模型，对召回结果重新排序。

这样做的整体目标，不是单纯增加查询数量，而是同时提升“表达清晰度、召回覆盖率和结果相关性”。

#### 一句总结版

Query 改写本质上是在用户问题和知识库文档之间做一层“检索适配”。它的目标不是改变用户意图，而是把用户意图转换成更适合召回的表达形式，从而提升 RAG 系统的检索效果和最终回答质量。

#### 延伸追问

如果面试官继续追问，可以从这几个方向展开：

- 改写效果怎么评估：可以看召回相关性、Top-K 命中情况、最终回答质量是否提升。
- 改写是否会引入偏差：会，所以要控制“改写不改意”，避免模型过度脑补。
- 什么时候不适合做复杂改写：当问题已经很清晰、或者延迟成本敏感时，可以只做轻量规范化。
- 为什么要和重排配合：因为 query 改写解决的是“怎么更好召回”，重排解决的是“召回后怎么把最相关结果放前面”。

### 2. 私有知识库召回效果应该从哪些方面优化

#### 核心理解

私有知识库的效果，不能简单理解为“把文档放进向量数据库就结束了”。真正影响召回质量的，往往是数据处理、切分策略、检索策略和重排策略的组合设计。

如果只做最基础的向量化入库，常见问题会包括：

- 文档切得过粗，导致召回内容噪声大
- 文档切得过细，导致上下文丢失
- 用户 query 和文档表达差异大，直接检索命中率低
- 多来源结果无法有效融合，重要内容排不到前面

#### 面试表达

如果面试官问我怎么提升私有知识库的召回效果，我通常不会只回答“换一个更强的向量模型”，而是会从完整链路来讲：  
第一是切分策略，第二是 query 改写，第三是召回层设计，第四是重排或融合策略，第五是文档组织方式。因为知识库效果本质上是系统问题，不是单点参数问题。

#### 常见优化点

##### 1. 切分策略优化

切分粒度直接决定召回效果。太粗会导致片段噪声大，太细又会破坏语义完整性。实际工程里通常会结合标题、章节、段落和语义边界来切，而不是只按固定 token 长度硬切。

##### 2. Query 改写

对口语化、模糊化问题先做规范化改写，必要时做多查询生成或复杂问题分解，降低 query 和文档表达方式不一致带来的召回损失。

##### 3. 分层召回

不要只依赖单一路径。常见做法是把向量召回、关键词召回、结构化过滤结合起来，先扩大候选集，再做精筛。这样比单一向量检索更稳。

##### 4. 重排与结果融合

召回后的结果通常还需要重排。可以引入 cross-encoder rerank，也可以用 `RRF`，也就是 `Reciprocal Rank Fusion`，把多路召回结果做倒数排序融合。它的价值在于综合不同检索路径的优势，降低单一路径偏差。

##### 5. 父子文档设计

有些场景适合把文档拆成“子块用于召回，父块用于补全文”。也就是先用细粒度子片段提高命中率，再返回更完整的父文档上下文，兼顾精度和可读性。

#### 一句总结版

私有知识库的召回优化，不是“换库”或者“换 embedding”这么简单，而是要从切分、改写、召回、重排和文档组织几个层面一起设计。

### 3. 怎么判断 RAG 的检索结果是否可靠

#### 核心理解

RAG 系统里，生成质量高度依赖检索质量。如果检索阶段拿回来的内容不相关、缺关键事实、来源过时或者多来源互相矛盾，那么模型即使生成得很流畅，也大概率是在“基于错误证据胡说”。

所以线上 RAG 不只是“能检索到东西”就够了，更重要的是判断“检索到的东西靠不靠谱”。

#### 不可靠结果的常见表现

- 召回内容和问题不相关
- 关键证据缺失
- 信息来源过时
- 多个来源之间内容冲突

#### 可靠性校验思路

##### 1. 离线评估

先构建评估集，用人工标注或标准答案评估检索质量。常见指标包括：

- `Hit@K`
- `MRR`
- `NDCG`

但不能只看相似度分数，因为相似度高不代表真的回答了问题，通常还需要人工抽样复核。

##### 2. 多来源一致性校验

对于 Top-K 结果，不只看单条最高分，还要看多个来源之间是否能互相印证。关键事实如果至少能被两个高可信来源支持，稳定性会更高；如果只有单一来源命中，就应该降权或标记低置信度。

##### 3. 来源可信度建模

不同来源不能一视同仁。官方文档、制度文件、产品手册、论坛帖子、个人博客的可信度本来就不同。工程上通常需要给来源打标签，并结合发布时间做衰减加权。

##### 4. 低置信度兜底

如果召回结果置信度低，可以按顺序做几层处理：

1. 先重排
2. 再改写 query 做二次检索
3. 仍然不可靠时触发拒答或保守回答

重点是不让模型在证据不足时硬答。

##### 5. 用户侧透明化

前端最好展示命中来源、时间、引用片段和置信度信息，并提供反馈入口，形成线上闭环。

#### 线上监控要看什么

除了命中率，还要看一些中间指标，例如：

- Top-K 的相关性分布
- 重排分数变化
- 用户点踩率
- 二次检索触发率
- 拒答率

因为很多检索跑偏问题，会先在这些中间指标上暴露出来。

#### 一句总结版

RAG 检索可靠性不是单点指标，而是一套“离线评估 + 来源校验 + 兜底机制 + 线上监控”的完整闭环。

### 4. 企业知识库应该怎么分层理解

#### 企业知识库的核心定位

企业知识库要解决的问题，不只是“文档问答”，而是把散落在聊天工具、协作文档、代码仓库、邮件、会议纪要等各个系统里的组织知识重新连接起来。

它和个人知识库最大的区别在于：企业知识库必须同时考虑权限、协作、审计、系统集成和规模化治理。

#### 两类常见部署方式

##### 1. 独立知识中枢模式

在现有业务系统之上搭建统一的 AI 搜索入口，不强制迁移所有数据，而是跨系统整合查询。

适合场景：

- 企业内部工具很多
- 系统之间数据分散
- 希望快速形成统一搜索入口

##### 2. 嵌入式 AI 模式

把 AI 能力直接嵌入企业原有办公系统或协作套件中，在原工作流内提供搜索、总结、生成能力。

适合场景：

- 企业已经深度使用某一套办公平台
- 希望降低使用门槛，减少平台切换

它的局限是，如果大量知识并不在那套系统里，覆盖面就会受限。

#### 自建知识库的三层能力

##### 1. 普通 RAG

把文档解析、切分、向量化后入库，基于检索结果生成回答。适合文档问答、手册查询、FAQ 等场景。优点是成熟、落地快；缺点是擅长“找资料”，不擅长处理复杂关系链路。

##### 2. Graph RAG

在 RAG 基础上增加知识图谱层，从文档中抽取实体和关系，支持基于关系链做查询和推理。适合金融、法律、风控这类关系复杂场景。缺点是建图和维护成本明显更高。

##### 3. Agent RAG

在前两者基础上引入 Agent，让系统能主动拆解问题、选择知识源、做多轮检索和综合推理。适合复杂研究、合规审查、多文档综合分析这类任务。缺点是速度更慢、成本更高、执行链路更难调试。

#### 面试表达

如果面试官问企业知识库怎么选型，我一般不会直接说“上 Agent RAG”，而是先看问题复杂度。  
简单问答用普通 RAG 就够了；需要跨文档关系推理时再考虑 Graph RAG；只有在任务本身需要多步拆解、多源整合、多轮检索时，才需要 Agent RAG。

#### 一句总结版

企业知识库的目标不是把所有资料都塞进 AI，而是让组织知识在需要时能够被找到、理解、验证和复用。

### 5. Agent 的规划能力到底在解决什么问题

#### 核心理解

Agent 的规划，本质上是把一个大目标拆成可执行的步骤序列。复杂任务如果没有规划，Agent 很容易在长链路执行里出现重复劳动、遗漏关键步骤、执行顺序混乱甚至中途跑偏。

所以规划能力不是为了“看起来更智能”，而是为了让复杂任务变得可拆解、可执行、可修正。

#### 常见规划范式

##### 1. Plan-then-Execute

先生成完整计划，再按步骤执行。优点是结构清晰，适合任务边界比较稳定的场景；缺点是计划一旦和环境变化不匹配，后续容易僵化。

##### 2. ReAct

每一步都先推理，再行动，再根据观察结果决定下一步。优点是更灵活，适合外部环境不确定、需要边执行边调整的任务；缺点是链路更长，也更容易引入推理噪声。

##### 3. 树状思考或任务分解

先把复杂问题拆成多个子问题，分别探索，再综合结果。适合需要比较、搜索、路线探索的复杂任务，但成本也更高。

#### 规划系统还需要什么配套能力

- 动态修正：执行报错、环境变化或中间结果异常时要能调整计划
- 错误恢复：不能一步失败就整条链路报废
- 质量评估：要看步骤是否合理、是否可执行、整体成本是否可接受

#### 工程评估指标

规划能力上线后不能只看“有没有输出计划”，更应该看：

- 规划修正次数
- 单步成功率
- 任务完成率
- 无效步骤占比

#### 一句总结版

真正有价值的 Agent 规划，不是生成一份好看的步骤清单，而是把规划做成一个“可执行、可修正、可评估”的子系统。

### 6. Agent 记忆不能简单等于“上下文 + 向量库”

#### 核心理解

很多人会把 Agent 记忆简单理解成“短期记忆就是上下文，长期记忆就是向量数据库”，这个说法太粗糙。更准确的理解是：短期记忆和长期记忆在存储位置、写入方式、读取方式、生命周期和治理要求上都不一样。

#### 短期记忆

短期记忆更像工作记忆，通常存在当前上下文窗口里，也就是 prompt 组织的一部分。它随着对话推进不断变化，主要服务当前任务，对话结束后通常就消失了。

适合放入短期记忆的内容：

- 当前任务状态
- 本轮对话中的临时变量
- 尚未沉淀为稳定事实的中间信息
- 敏感且不应长期保留的片段

#### 长期记忆

长期记忆通常放在外部存储里，可跨会话存在，常见形态包括：

- 向量记忆
- 图谱记忆
- 键值记忆
- 事件流记忆

它不是自动把整段聊天原样存进去，而是应该先做事实抽取和结构化压缩，再决定是否写入。

适合放入长期记忆的内容：

- 用户画像
- 稳定偏好
- 项目背景事实
- 可复用的跨会话信息

#### 长期记忆写入与召回流程

比较合理的流程是：

1. 从对话中抽取稳定事实
2. 压缩成结构化条目
3. 生成 embedding 或结构化索引
4. 写入用户隔离的记忆空间
5. 检索时先召回，再重排，只把高相关条目拼回 prompt

这样做比存原始流水更干净，也更利于后期治理。

#### 多会话记忆治理

长期记忆如果不治理，时间一长会严重污染检索效果。常见治理要求包括：

- 每个用户独立命名空间
- 禁止跨用户召回
- 敏感信息脱敏或加密
- 合并重复事实
- 清理过期偏好

#### 一句总结版

Agent 记忆的本质不是“存聊天记录”，而是把有价值、可复用、可治理的信息沉淀成一个可写入、可检索、可维护的系统。

### 7. 企业级 AIGC / Agent 项目要具备哪些核心能力

#### 核心理解

企业级 AIGC 项目不能只停留在“调通模型接口”或“做出一个 Demo”。真正能上线的项目，通常至少要覆盖数据处理、召回优化、在线指标、数据反馈和安全治理这些核心层面。

#### 核心能力框架

##### 1. 多模态数据处理

要能处理 Word、PPT、PDF 等非纯文本数据，明确文档解析、文本抽取、切分规则和索引构建方式。

##### 2. 检索与召回优化

要从切分、Query 改写、分层召回、重排、父子文档设计等多个维度提升知识库效果，而不是只依赖单一向量检索。

##### 3. 在线核心指标体系

项目上线后至少要关注这些指标：

- `QPS`
- `TTFT`，即首个 token 返回时间
- 检索命中相关指标
- 幻觉率
- 拒答率
- 安全审核命中率

这些指标决定系统是否具备真实可用性。

##### 4. 数据飞轮

上线后不能停在初版效果上，而要基于真实用户数据持续做：

- 指标评估
- Bad case 分析
- Prompt / 检索 / 流程优化
- 必要时对底层模型做微调或后处理优化

##### 5. 数据回流与反馈机制

需要打通线上数据回收路径，建立用户反馈、人工审核、安全审核和问题追踪机制，形成闭环。

#### 面试表达

如果面试官问我怎么看一个 Agent 项目是不是工业级，我通常会看它有没有完整闭环：  
前面有没有把数据和召回做好，中间有没有指标体系和安全机制，后面有没有反馈回流和持续优化。如果缺少后半段闭环，这类项目通常更像 Demo，而不是上线系统。

#### 一句总结版

企业级 AIGC / Agent 项目的核心，不是模型本身有多强，而是有没有形成“数据处理 - 检索优化 - 指标监控 - 反馈回流 - 持续迭代”的工程闭环。

### 8. 合格的 AIGC 开发工程师需要哪些能力

#### 能力结构

##### 1. 大模型基础认知

需要理解神经网络、Transformer、大模型基础架构，知道上下文窗口为什么受限，理解应用开发侧真正需要掌握的底层边界。应用开发岗位不一定要深挖强化学习，但要对模型能力边界有基本判断。

##### 2. 企业级落地能力

除了会调模型接口，更要能解决这些工程问题：

- 短期记忆与长期记忆设计
- 向量库与知识图谱的合理使用
- Agent 的反思、自检、试错机制
- 多模态接入后的性能问题
- 首字延迟与并发能力优化
- 缓存机制设计
- 恶意输入与安全防护
- 外部工具与技能编排

##### 3. 上线能力

真正完整的能力还包括：

- 灰度发布
- A/B 测试
- 回滚预案
- 线上监控与报警

因为项目只有真正上线跑通，很多工程问题才会暴露出来。

#### 一句总结版

合格的 AIGC 工程师，不只是“懂模型”，更是要能把模型能力稳定地接进真实业务系统，并对效果、成本、安全和上线结果负责。

### 9. 大模型落地为什么经常卡在工程化而不是模型本身

#### 核心理解

很多大模型项目真正的难点不在“模型能不能回答”，而在组织推进、知识治理、成本控制、安全合规和存量系统兼容这些工程问题。

#### 常见工程化难点

##### 1. 治理框架

不能只有技术方案，还要明确责任边界、审核节点、反馈机制，否则上线后很容易权责不清。

##### 2. 成本控制

不是所有请求都该走最强模型。常见做法是增加路由层，先判断任务复杂度，简单任务走轻量模型，复杂任务走高性能模型，降低整体推理成本。

##### 3. 知识治理

知识库质量直接决定上限。需要建立知识准入标准、质量评分、定期清理和隐性知识显性化机制，否则知识越多，噪声可能越大。

##### 4. 多智能体协作

复杂业务流程中，多个 Agent 最好职责单一、边界清晰，通过调度层和消息机制协作，而不是把所有能力堆进一个大 Agent 里。

##### 5. 安全合规

需要对权限、脱敏、访问审计、数据出网进行严格管控，尤其是涉及企业敏感数据时。

##### 6. 存量系统对接

真实企业往往有老系统、无 API 系统或难改造系统，这时候可能需要用 `RPA` 之类方式桥接，而不能假设所有系统都能标准接入。

#### 一句总结版

大模型项目从 Demo 到上线，最大的鸿沟通常不在模型调用，而在工程治理和组织落地。

### 10. 什么样的 Agent 项目更像真实可深挖的项目

#### 核心判断标准

如果一个场景只靠一次普通对话调用就能完成，那它未必适合包装成 Agent 项目。真正值得深挖的 Agent 场景，通常有这几个特征：

- 需要多步推理
- 需要跨多数据源整合信息
- 需要处理异步任务
- 需要根据环境变化动态调整执行路径

#### 面试里常见追问点

##### 1. 选型是否合理

不是技术堆得越多越好，而是要和场景匹配。比如只有几千条数据，却上高复杂度分布式向量数据库，这种选型就容易被质疑。

##### 2. 工程实现是否讲得清

面试官往往会追问：

- 上下文怎么分层管理
- 记忆什么时候写入
- 噪音怎么过滤
- Prompt 怎么设计
- 工具调用怎么编排

##### 3. 效果评估是否完整

项目不能只说“效果不错”或者“准确率提升了”，还要说明：

- 测试集怎么构建
- 覆盖了哪些 case
- 指标怎么定义
- 是否有自动化回归

#### 术语纠正提醒

- `Scratch` 更准确的说法通常是 `Scratchpad`，指 Agent 执行过程中用于中间推理或中间状态记录的工作区。
- `MCP 函数` 更准确地说可以表述为 `MCP 工具调用` 或 `MCP tools`，强调的是模型通过标准协议访问外部工具的能力。

#### 一句总结版

一个 Agent 项目是否足够扎实，关键不在名字里有没有 “Agent”，而在于它是否真的需要多步决策、多源协同和完整的工程闭环。
