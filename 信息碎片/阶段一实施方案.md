# 1. 文档目的

本文档用于明确光伏智能问答系统第一阶段的建设目标、范围边界、技术方案、任务分工、验收标准与升级预留策略，确保：

1. 第一阶段能够按期交付一个可演示、可试用、可验收的 Web MVP。
2. 第一阶段的架构设计能够满足既定验收标准。
3. 第一阶段的实现方式不会成为第二阶段系统升级的阻碍，能够快速平滑演进到“工具融合 Function Calling”架构。

# 2. 项目整体三阶段路线

## 2.1 阶段一：文档知识库 RAG
目标是围绕现有文档库与光伏运维故障案例库，建设一个基于文档检索增强生成的智能问答系统，实现：

- 文档解析入库
- 语义检索与关键词混合检索
- 基于证据的答案生成
- 来源引用与拒答机制
- Web 端问答 MVP

## 2.2 阶段二：工具融合 Function Calling
在第一阶段之上新增系统工具能力，实现：

- 数据库查询
- API 调用
- SCADA / DCS 接口接入
- 工单系统接入
- RAG 与实时系统的组合调用
- 多轮会话中的工具路由与状态管理

## 2.3 阶段三：推理智能体 Reasoning Agent
在第二阶段之上新增规划、推理、执行和分析能力，实现：

- 异常检测
- 根因分析
- 时序建模
- 主动预警
- 自动分析报告生成

# 3. 第一阶段建设结论

当前第一阶段已明确如下边界：

## 3.1 已确认决策
- 第一阶段只做 Web 前端 MVP
- 向量库使用 `pgvector`
- 初期知识源仅接入：
  - 现有文档库
  - 光伏运维故障案例库

## 3.2 第一阶段建设目标
第一阶段不追求做“全能 AI 助手”，只聚焦做成一个稳定、可信、可追溯的文档问答系统，优先解决：

- 文档能否顺利入库
- 问题能否基于知识库回答
- 回答是否引用来源
- 无依据时是否拒答
- Web 页面是否形成可用闭环

## 3.3 第一阶段不纳入范围
以下内容明确不在本阶段开发范围内：

- 移动端页面
- 语音输入
- 实时数据库查询
- 实时 API 调用
- SCADA / DCS 接入
- 工单系统对接
- 推理 Agent
- 时序建模
- 主动预警
- 全量 OCR 难文档治理
- 复杂扫描 PDF、接线图、图片表格精细解析
- 多租户隔离
- 完整后台运营平台

# 4. 第一阶段业务目标

## 4.1 核心目标
围绕“光伏领域文档知识问答”建立第一版 MVP，满足以下业务价值：

1. 让业务用户通过自然语言提问，快速获得与光伏运维、设备故障、巡检、发电量分析相关的答案。
2. 所有回答必须尽可能基于知识库证据输出，减少幻觉。
3. 系统面对无关问题、证据不足问题时，不乱答，统一输出兜底话术。
4. 为第二阶段接入实时业务工具打下稳定底座。

## 4.2 MVP 成功标志
第一阶段成功的标志不是“页面做完”，而是以下闭环成立：

- 管理员可导入文档
- 文档可解析、切块、向量化、入库
- 用户可在 Web 页面发起问答
- 系统可返回基于文档证据的答案
- 答案带来源引用
- 无依据时拒答
- 日志与未命中问题可追踪

# 7.3 核心设计原则

## 7.3.1 证据优先
所有回答都必须优先基于知识库片段，不允许无依据扩写。

## 7.3.2 拒答优于幻觉
第一阶段的质量核心不是“尽量都回答”，而是“答得准，答不准就拒答”。

## 7.3.3 模块分层清晰
第一阶段就要按可升级架构分层，避免第二阶段重写。

## 7.3.4 预留工具扩展口
问答服务不与 RAG 强耦合，后续可以在第二阶段增加工具路由器。

# 8. 第一阶段核心能力设计

# 8.1 文档入库链路

```text
上传文件
  -> 计算 SHA256 去重
  -> 创建 parse_task
  -> 选择解析器
  -> 解析原文
  -> 文本清洗
  -> 分块
  -> 向量化
  -> 写入 kb_document / kb_document_segment
  -> 更新任务状态
```

## 8.1.1 第一阶段建议支持格式优先级
按以下优先级实现：

1. TXT / MD
2. DOCX
3. 文本型 PDF

理由：
- 最快形成有效闭环
- 最适合当前样本文档库
- 可以满足第一阶段验收
- 不阻塞第二阶段架构升级

---

# 8.2 文本切块策略

## 8.2.1 推荐策略
采用递归语义切块，不使用死板固定窗口：

- 目标块大小：600~800 字
- overlap：60~100 字
- 按标题、段落、换行优先切分
- 语义断裂时才回退到句号或长度窗口切分

## 8.2.2 这样设计的原因
- 更适合说明书、案例库、巡检文档
- 有助于提高召回质量
- 更利于第二阶段继续复用同一知识切块结果

---

# 8.3 检索策略

## 8.3.1 第一阶段采用混合检索
推荐第一阶段直接采用：

- 向量检索：负责语义召回
- 关键词检索：负责术语精确匹配
- 融合排序：综合得分输出 Top-K

## 8.3.2 不建议第一阶段只做纯向量检索
原因：
- 光伏领域中设备名、故障码、专业术语很多
- 纯向量检索容易发生语义漂移
- 第一阶段验收指标对召回率和忠实度要求较高

## 8.3.3 第一阶段建议返回
- 默认返回 Top-5 引用片段
- 支持来源映射到：
  - 文档名
  - 页码 / 章节 / 段落
  - 片段内容摘要

---

# 8.4 问答策略

## 8.4.1 回答类型
第一阶段建议保留 4 种回答类型：

- `faq`
- `rag`
- `none`
- `fallback`

说明：
第一阶段不强依赖通用聊天能力，避免偏题。

## 8.4.2 回答规则
- 仅在证据足够时输出正常回答
- 证据不足只回答已知部分
- 对敏感问题没有证据时必须拒答
- 无关问题返回固定兜底话术
- LLM 失败时使用模板式证据拼接兜底

---

# 8.5 固定兜底话术

```text
您好！您提出的问题超出了我的当前能力范围。作为一名专注于光伏电站监测与数据分析的智能助手，我主要擅长处理与发电量、设备告警、运维管理等相关的问题。现在，我可以继续为您解答有关电站的问题吗？
```

此话术满足当前产品要求，并且后续可在第二阶段继续复用。

- `ref_type`
- `ref_source`
- `ref_metadata_json`

意义：第二阶段可以同时引用：
- 文档片段
- 数据库查询结果
- API 返回结果
- 工具调用结果

这一步是“快速升级第二阶段”的关键。

# 10. 第一阶段接口设计原则

# 10.1 API 按模块拆分
建议按以下模块拆分接口：

- `/api/auth/*`
- `/api/qa/*`
- `/api/documents/*`
- `/api/faq/*`
- `/api/logs/*`
- `/api/sessions/*`

这样有利于第二阶段新增：
- `/api/tools/*`
- `/api/agent/*`

---

# 10.2 QA 接口不要与 RAG 强耦合
第一阶段问答接口建议定义为：

- `ask_question()`
- 返回统一结构：
  - answer
  - answer_type
  - references
  - confidence
  - trace_id
  - debug_info（可选）

不要将接口命名成“只支持文档 RAG”的强耦合名字，否则第二阶段扩展时会被迫改前端调用链。

---

# 10.3 第二阶段升级预留建议
第一阶段 API 返回结构中建议预留以下字段：

- `route`
- `tool_calls`
- `citations`
- `decision`
- `fallback_reason`

即使第一阶段先不填全，也能保证第二阶段升级时前端无需大改。

# 14. 第一阶段验收标准

# 14.1 功能验收标准

必须满足以下项目：

- 可上传现有文档与故障案例文档
- 文档可解析、切块、向量化、入库
- 用户可通过 Web 页面发起提问
- 系统可返回问题答案
- 每条可回答结果必须带来源引用
- 无依据问题必须拒答或兜底
- 用户可查看历史会话
- 管理员可查看文档状态
- 管理员可查看未命中问题
- 管理员可查看基础问答日志

---

# 14.2 质量验收标准

依据阶段图要求，第一阶段建议使用如下指标：

## 检索质量
- `Recall@5 >= 85%`

## 回答忠实度
- `Faithfulness >= 90%`

## 回答相关性
- `Answer Relevance >= 0.80`

## 来源可追溯性
- `引用来源覆盖率 = 100%`

---

# 14.3 第一阶段验收自检结论

基于本文档当前方案，第一阶段是可以满足验收标准的，原因如下：

## 对 Recall@5 的支持
本方案采用：
- 语义切块
- 向量检索
- 关键词检索
- Top-K 引用
- golden QA 调参

这比纯向量检索更有机会达到 `Recall@5 >= 85%`。

## 对 Faithfulness 的支持
本方案明确规定：
- 严格基于证据回答
- 证据不足必须拒答
- LLM 失败可退回模板式证据答案
- 不把“通用聊天”作为第一阶段主路径

因此更有利于实现 `Faithfulness >= 90%`。

## 对 Answer Relevance 的支持
本方案通过：
- FAQ 优先
- 混合检索
- RAG 主链路
- 无关问题拒答

可以有效减少答非所问，更容易达到 `>= 0.80`。

## 对引用来源率的支持
本方案从表结构到接口返回都将来源引用作为必备能力，而不是可选能力，因此可满足 `100%` 的来源追溯要求。

---

# 14.4 验收前提条件说明
虽然方案本身满足验收目标，但要真正达到指标，必须补齐以下执行动作：

1. 构建 `100~200` 条 golden QA 集
2. 用真实样本文档做至少一轮调参
3. 对低质量文档先做筛选
4. 对兜底逻辑和阈值进行回归验证

换句话说：方案具备达标条件，但验收通过依赖执行质量，不是只靠架构名称。

# 16. 第二阶段快速升级兼容性设计

# 16.1 结论
本文档方案可以快速接入第二阶段升级，前提是严格按“分层 + 统一问答入口 + 预留引用结构 + 预留工具路由字段”实施。

---

# 16.2 为什么这套方案适合快速升级

## 16.2.1 QA 入口统一
第一阶段就把问答入口定义为统一的 `ask_question()`，而不是写死成“只做 RAG”的入口。

这样第二阶段只需要把内部链路升级成：

```text
ask_question()
  -> intent/router
  -> RAG or tool calls or both
  -> answer assembly
```

前端几乎不用改。

## 16.2.2 引用结构可扩展
第一阶段如果把 `qa_reference` 设计成通用引用结构，第二阶段就可以支持同时引用：

- 文档片段
- 数据库查询结果
- 实时 API 数据
- 告警系统结果
- 工单系统结果

不需要重建整个回答展示层。

## 16.2.3 集成层独立
第一阶段把解析器、Embedding、LLM 放在独立适配层，第二阶段加工具时只需新增：

- `tool_registry`
- `tool_executor`
- `tool_router`

不需要重写知识库层和问答页。

## 16.2.4 日志模型可扩展
如果 `qa_record` 已预留 `decision_json`、`trace_id`、`latency_ms`，第二阶段就能直接记录：
- 选了哪个工具
- 提取了哪些参数
- 调用了几次工具
- 哪一步失败了

这对二阶段调试至关重要。

---

# 16.3 第二阶段升级时可直接复用的内容

第一阶段完成后，以下模块可直接复用：

- 前端问答页
- 会话页
- 文档管理页
- 文档库与向量库
- FAQ 能力
- 检索服务
- 回答展示层
- trace_id 日志链路
- 拒答与兜底话术
- 用户权限骨架

---

# 16.4 第二阶段仅需新增的能力

在第一阶段之上新增：

- 工具注册表
- 工具调用协议
- 参数抽取
- 工具路由器
- 会话状态增强
- RAG 与工具的混合决策
- 超时 / 降级策略增强

这说明第一阶段不会推翻重做，而是“增量升级”。

---

# 16.5 第二阶段升级建议接口预留

建议第一阶段就预留这些响应字段：

```json
{
  "answer": "...",
  "answer_type": "rag",
  "references": [],
  "confidence": 0.92,
  "trace_id": "xxx",
  "route": "rag",
  "tool_calls": [],
  "decision": {},
  "fallback_reason": null
}
```

即使第一阶段只使用其中一部分，也可以显著降低第二阶段改造成本。

# 17. 风险与应对

# 17.1 文档质量不稳定
## 风险
DOCX / PDF 结构不稳定，解析异常会影响入库。

## 应对
- 先选高质量样本文档建首批库
- 对异常文档标记失败，不阻塞主流程
- 保留解析失败日志
- 第二阶段再系统补 OCR / 扫描件治理

---

# 17.2 检索召回不稳定
## 风险
纯向量或纯关键词都可能效果不稳定。

## 应对
- 第一阶段直接上混合检索
- 构建 golden QA
- 周期性调参
- 对专业术语建立关键词增强策略

---

# 17.3 回答幻觉
## 风险
模型可能补全文档中没有的信息。

## 应对
- 严格 Prompt 约束
- 证据不足必须拒答
- 敏感问题提高准入阈值
- LLM 异常时模板兜底

---

# 17.4 两人开发互相阻塞
## 风险
接口不稳定会拖慢联调。

## 应对
- 第 1 周先冻结 API 契约
- 前端先用 mock 数据推进
- 所有接口变更必须同步
- PR 小步快跑

---

1. Web MVP 可运行

3. 用户可提问并获得带来源引用的回答
4. 无关或无依据问题能正确拒答

6. 建立了 `100~200` 条 golden QA 集

8. 主分支代码可部署、可演示、可回归
9. 接口与数据结构已为第二阶段预留升级空间

# 20. 最终结论

## 20.1 对第一阶段验收的结论
本方案在范围控制、技术选型、任务拆分、数据模型和质量控制上，能够支撑第一阶段验收标准，前提是按本文档要求完成：

- 混合检索
- 来源引用
- 拒答机制
- golden QA 测试集
- 调参与回归测试

因此，从方案层面判断：**可以满足第一阶段验收要求**。

## 20.2 对第二阶段升级的结论

因此，从架构演进层面判断：**可以快速接入第二阶段系统升级**。
